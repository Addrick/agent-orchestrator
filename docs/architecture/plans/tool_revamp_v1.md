## Goal

Lift the Phase C policy in `chat_system.stream_response` that refuses any persona with `has_tools_enabled=True`. Portal clients (and any future streaming consumer) should be able to use tool-enabled personas with a linear event stream — token deltas interleaved with tool-call lifecycle events — and no drain-and-restart.

This is the **mechanics** track. Capability classification, composition rules, per-tool gating, and meta-agent sandboxing live in the sibling plan `tool_security_framework.md` and are not preconditions for shipping this.

## Why

Today the portal cannot run any persona that has tools enabled. That's the only remaining functional gap between portal personas and Discord/Gmail personas. Designs already exist in `portal_engine_reintegration.md` (lines 85–90):

- Tool-call-in-stream as additional event types
- Portal renders them as `<details>` thought-process blocks inline between `TokenEvent` runs
- Linear stream, no drain-and-restart

The blocker has been the open ToolLoop extraction (`plans/toolloop_extraction.md`) and an assumed dependency on a security framework. Both can be sidestepped: extract a stream-shaped loop now, treat the inbound tool list as already-filtered (whoever calls the loop decides what's in it), and ship the security model as a separate layer that plugs in later without reshaping the event surface.

## Non-goals

- **Capability classification** (read/write × local/online axes) — sibling plan
- **Composition rules** ("if `web_search` then deny `local_write`") — sibling plan
- **Per-persona allow/ask/deny profiles** — sibling plan
- **Meta-agent / sub-agent sandboxing** — sibling plan, no agent currently spawns agents
- **Per-agent tool subsets** — orthogonal; agents have working hardcoded pipelines
- **MCP client** — gated behind security framework per `mcp_strategy.md` decision #1
- **Streaming on Discord/Gmail interfaces** — already deliberately parked in `portal_engine_reintegration.md`

## Design

### New event types

Add to whatever module currently defines `TokenEvent` / `DoneEvent` / `ErrorEvent` (`src/chat_system.py` or its events module):

```python
@dataclass
class ToolCallStartEvent:
    tool_name: str
    arguments: Dict[str, Any]
    call_id: str  # opaque id, lets portal pair start↔result if multiple calls overlap

@dataclass
class ToolCallResultEvent:
    call_id: str
    tool_name: str
    result: str          # serialized result the LLM will see
    error: Optional[str] # populated on tool execution failure; result still set to a fallback string for the LLM
```

`call_id` is generated by the loop, opaque to consumers. Portal uses it to fold each `<details>` open/close pair correctly even if a future provider streams parallel tool calls.

### ToolLoop extraction (event-yielding shape)

`src/tools/tool_loop.py` (new):

```python
class ToolLoop:
    def __init__(self, text_engine, tool_manager, max_iterations=5): ...

    async def run(
        self,
        persona,
        messages,
        params,
        tools,                # already-filtered list; loop trusts caller
        service_context=None, # passed through to tool_manager.execute_tool
    ) -> AsyncIterator[GenerationEvent]:
        # 1. Call text_engine.stream_messages(persona, messages, params, tools)
        # 2. Forward TokenEvent unchanged
        # 3. On tool call detected in stream: yield ToolCallStartEvent,
        #    await tool_manager.execute_tool, yield ToolCallResultEvent,
        #    append the call+result to messages, loop.
        # 4. Cap at max_iterations; final iteration yields DoneEvent.
        # 5. Errors yield ErrorEvent and terminate.
```

Two principles:
- **Loop trusts the tool list it's handed.** Filtering happens in the caller (engine maps persona → tools). Sibling plan will replace that mapping with a richer policy without touching the loop.
- **Linear stream.** No drain-and-restart, no nested generators. Tool calls are events, not interrupts.

The existing `_orchestrate` kernel in `ChatSystem` already runs `text_engine.stream_messages` directly. This phase moves the iteration logic out of `_orchestrate` into `ToolLoop.run`, and `_orchestrate` becomes a thinner wrapper that just calls `tool_loop.run(...)` and forwards events.

### Lift the stream_response guard

In `chat_system.stream_response`, remove the `has_tools_enabled` refusal. The kernel now handles tools natively.

### Portal rendering

`/v1/chat/completions` SSE relay (already shipped Phase D) extends to:

- `ToolCallStartEvent` → emit a custom SSE frame (e.g. `event: derpr-tool-start`) with `{tool_name, arguments, call_id}`. Portal opens a `<details><summary>tool_name</summary>` block, renders args.
- `ToolCallResultEvent` → emit `event: derpr-tool-result` with `{call_id, result, error}`. Portal closes the matching block, renders result/error.
- `TokenEvent`, `DoneEvent`, `ErrorEvent` already handled.

OAI-compatible clients that don't understand `derpr-tool-*` frames simply skip them — text deltas continue to render normally.

### Tool execution + retry semantics

No changes to `tool_manager.execute_tool` signature, `WRITE_TOOLS` gating, or CONFIRM-mode emoji confirmation. The portal currently runs in autonomous mode (no emoji surface), so write-tool gating in CONFIRM mode is irrelevant for the immediate use case. If/when CONFIRM-mode personas hit the portal, that's a separate UX problem — punt.

## Test strategy

- **Phase 2:** unit tests for `ToolLoop.run` — single tool call, multiple sequential calls, error path, max-iterations cap. Mock `text_engine.stream_messages` and `tool_manager.execute_tool`.
- **Phase 3:** integration test for `chat_system.stream_response` with a tool-enabled persona — verify event order is `TokenEvent* → ToolCallStartEvent → ToolCallResultEvent → TokenEvent* → DoneEvent`.
- **Phase 3:** portal adapter test — `_make_real_adapter` fixture (already exists post-Phase D), assert SSE frames serialize tool events correctly.
- **Phase 4:** manual portal smoke (caveman doesn't auto-test browsers) — load a tool-enabled persona, watch a `<details>` block render.

## Security posture (deliberately minimal here)

This plan **does not** add new security primitives. It preserves what exists:

- `enabled_tools` filter on persona (passed to engine, then to loop)
- `service_bindings` filter via `ServiceIntegration.register_tools`
- `WRITE_TOOLS` set + CONFIRM-mode emoji gating (unchanged; portal runs autonomous)

The invariant to write down now (and enforce in the sibling plan): **no persona may compose `network_read`-tagged tools with `local_write`-tagged tools without explicit policy override.** Today no portal-reachable persona has any `local_write`-class tool, so the invariant holds vacuously. Sibling plan formalizes the tags + check.
